# 🗺️ 通路戰情地圖系統 — 需求與產品規格文件 (PRD)

本文件整理並簡化了「通路戰情地圖系統 (Channel War Map)」的核心需求，旨在為團隊與工程師提供清晰的產品範疇與架構藍圖。

---

## 🎯 一、 產品目的與定位
整合實體通路運營四大維度：**位置**、**庫存**、**效期**、**客情（拜訪與叫貨）**。以地圖視覺化為核心，協助業務代表提升巡店效率、降低產品過期損失，並幫助業務主管掌握全局戰況、輔助營運決策。

---

## 👥 二、 核心使用者與價值主張

| 使用者角色 | 關注核心問題 | 系統提供的核心價值 |
| :--- | :--- | :--- |
| **業務代表 (Sales Rep)** | • 今天該去哪裡補貨？<br>• 哪家店有過期風險？<br>• 哪些通路太久沒拜訪？ | **提高巡店人效與防損**：<br>透過地圖與顏色警示，一秒篩選高風險與高價值通路，精準安排每日拜訪路線。 |

## 🧭 三、 四大核心戰術地圖圖層
系統核心功能是以「地圖」為載體，切換不同的業務圖層，地圖標記 (Marker) 依據以下商業規則動態變更顏色：

### 1. 🏪 營運通路狀態圖層 (預設圖層)
依據合作進度顯示通路屬性：
*   🔵 **合作通路**：已正式簽約合作的通路。
*   🟡 **洽談中通路**：正在接洽、提案中的潛力客戶。
*   🟢 **待開發通路**：已搜集名單、計畫前往開發的新據點。
*   🔴 **已停止合作**：因合約到期或客情因素暫停合作的通路。

### 2. 📦 庫存分析圖層 (補貨地圖)
比對「目前庫存量」與「安全庫存量」，提供補貨警示：
*   🟢 **庫存正常** (現存量 $\ge$ 安全庫存)：無缺貨風險。
*   🟡 **低水位** (現存量 $<$ 安全庫存)：建議安排補貨。
*   🔴 **無庫存** (現存量 $=$ 0)：**緊急缺貨告警**，優先安排拜訪。

### 3. ⏳ 效期管理圖層 (防損地圖)
計算通路內所有商品「最快到期日」之剩餘效期：
*   🟢 **安全** (剩餘效期 $>$ 12 個月)：無過期風險。
*   🟡 **臨期警告** (剩餘效期 6~12 個月)：需要安排促銷、陳列調整或換貨。
*   🔴 **急迫過期** (剩餘效期 $<$ 6 個月)：**高過期風險**，需立即介入處理。

### 4. ❄️ 叫貨頻率圖層 (客戶溫度地圖)
計算自「最後一次叫貨日」至今累積的天數，偵測客情溫度：
*   🟢 **正常** (0~30 天)：往來頻繁，客情熱絡。
*   🔵 **關注** (31~60 天)：叫貨間隔拉長，需主動致電維繫。
*   🟡 **警示** (61~90 天)：可能被競品滲透，列為預警回訪目標。
*   🟣 **失溫通路** (90 天以上)：**面臨流失風險**，列為急需挽回目標。

## 📈 四、 潛力分級評分機制 (A / B / C 級)
系統依據**月銷售額**、**年成長率**、**叫貨頻率**與**商品數量**，自動對通路進行潛力權重打分：
*   🏆 **A 級通路**：高銷售、高成長、高回購的明星客戶，應列為重點維護對象。
*   🥈 **B 級通路**：穩定叫貨的中流砥柱，提供標準維護流程。
*   🥉 **C 級通路**：低活躍度或剛起步的通路，需評估開發潛力或調整拜訪頻次。

## 📝 五、 實體欄位定義 (Database Schema Guidelines)

### 1. 通路主表 (`channels`)
*   `id`: UUID (主鍵)
*   `name`: 字串 (通路名稱)
*   `type`: 字串 (合作中 / 洽談中 / 待開發 / 終止合作)
*   `potential_level`: 字串 (A / B / C)
*   `address`: 字串 (地址)
*   `latitude`: 浮點數 (經度)
*   `longitude`: 浮點數 (緯度)
*   `contact_person`: 字串 (聯絡人)
*   `phone`: 字串 (連絡電話)
*   `owner_sales`: 字串 (負責業務)
*   `cooperate_start_date`: 日期 (合作開始日期)
*   `remarks`: 文字 (備註說明)

### 2. 庫存與商品明細表 (`inventory_items`)
*   `id`: UUID (主鍵)
*   `channel_id`: UUID (關聯至 channels)
*   `product_name`: 字串 (商品名稱)
*   `current_stock`: 整數 (目前庫存量)
*   `safety_stock`: 整數 (安全庫存量)
*   `expiry_date`: 日期 (商品到期日)
*   `last_updated`: 日期 (最後更新日期)

### 3. 拜訪紀錄表 (`visit_records`)
*   `id`: UUID (主鍵)
*   `channel_id`: UUID (關聯至 channels)
*   `visit_date`: 日期 (拜訪日期)
*   `visitor`: 字串 (拜訪人員)
*   `summary`: 文字 (內容摘要)
*   `todo_items`: 文字 (待辦事項)

## 🛠️ 七、 技術架構與 SaaS 擴充可行性評估
對於工程師提出的架構詢問，以下為建議方案：

1.  **為什麼選擇 Supabase + Next.js + Mapbox？**
    *   **極速開發 (Fast MVP)**：Next.js App Router 具備一流的前端效能與 API Routing；Supabase (PostgreSQL) 內建強大的實時機制與安全存取 (RLS)，能在數天內搭建出商用級別的後端。
    *   **地圖極致渲染**：Mapbox 具有超高性能的向量地圖渲染，在大數據點標記時依然滑順，且自訂圖層樣式極為便利。
2.  **SaaS 擴充可行性 (SaaS Ready)**：
    *   **資料隔離**：在 Supabase 中可建立 `tenant_id`，利用 **Row Level Security (RLS)**，讓不同企業租戶的資料在資料庫底層徹底隔離。
    *   **權限管控**：可以設計 Roles（管理者、區域經理、第一線業務），限制資料檢視範圍。
    *   **無縫升級**：未來可整合 Stripe 訂閱支付、加載 LINE Bot 模組或透過 API 與企業內部的 SAP/Oracle ERP 系統進行定時庫存同步。
