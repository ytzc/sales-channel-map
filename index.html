🗺️ 通路戰情地圖系統

本系統將實體通路運營的四大維度：**位置**、**庫存**、**效期**、**客情（拜訪與叫貨）**高度整合，以地圖視覺化為核心，解決「資訊孤島」、「過期損耗」與「巡店效率低」等痛點。

傳統藥局、診所等實體通路管理缺乏整合。本系統以「地圖」為載體，將營運指標與地理位置相結合，避免資訊孤島，提升決策效率。

📦 2. 庫存分析圖層 (補貨地圖)

STOCK_STATUS

系統比對「在庫總數」與「安全水位」之差值，即時生成補貨通知：

🟢 庫存正常
庫存量 ≥ 安全水位
🟡 低於安全水位
庫存量 < 安全水位
🔴 緊急缺貨

⏳ 3. 效期追蹤圖層 (防損地圖)

EXPIRY_STATUS

追蹤通路中所有庫存商品「最快到期日」之剩餘效期，進行防過期處理：

🟢 安全
剩餘效期 > 12 個月
🟡 臨期警告
剩餘效期 6 ~ 12 個月
🔴 急迫過期
剩餘效期 < 6 個月 (立即調配或回收)

❄️ 4. 叫貨頻率圖層 (客戶溫度地圖)

ORDER_TEMP

依據店家「最後一次叫貨日」與今天相差之天數，研判客情溫度，防止流失：

🟢 正常
0 ~ 30 天
🔵 關注
31 ~ 60 天
🟡 警示
61 ~ 90 天
🟣 失溫
90 天以上 (極高危急)

📈 通路潛力分級評分機制 (A / B / C 級)

系統權重自動運算

系統整合**月銷售額**、**年成長率**、**叫貨頻率**與**庫存商品品項數**，自動進行權重評分，並在戰術地圖上將潛力客戶標示分級：

🏆A 級通路

高銷售、高成長、高叫貨頻率的「明星客戶」。業務經理應列為 VIP 巡檢對象，確保競品無法入侵。

🥈B 級通路

銷售穩定、回購週期固定的「主力客戶」。提供標準客情維護，維持健康庫存與固定巡迴拜訪。

🥉C 級通路

低銷售、低活躍或新成立的「潛在開發點」。建議降低拜訪頻率改以電話客服代替，節省人力成本。

🛠️ 一、 技術架構選擇評估與工程建議

⚡Next.js 14 (App Router)

具有高擴充性的 React 首選框架。透過其靜態導出 (Static HTML Export) 功能，可完美相容於 GitHub Pages 等純靜態伺服器；未來可平滑升級為 Node.js 動態 Server 生態。

🔋Supabase (PostgreSQL)

業界首創的 Serverless PostgreSQL。其內建的 Row-Level Security (RLS) 權限防禦機制，能直接滿足「業務僅能看自轄通路、主管能看全局」的資料安全性要求。

🌍Mapbox GL JS

基於 Web-GL 的高效地圖引擎。在台北市載入數千個通路標記點時，能維持每秒 60 影格流暢縮放不卡頓。自訂顏色主題極其精美，適合戰情大螢幕呈現。

🚀 二、 未來擴充 SaaS 之階段路徑 (Roadmap)

1

階段一：產品概念驗證 MVP (當前靜態純網頁)

已就緒 (開箱即測)

全網頁以純靜態 HTML/CSS + Mock Data 高模擬展現四大圖層與拜訪紀錄。免去跟公司舊 ERP 或雲端資料庫串接之繁瑣手續，提供業務代表和主管最直覺的操作試用與改進依歸。

2

階段二：Supabase 串接與第一線定位打卡

第二階段 TODO

對接 Supabase 雲端即時 PostgreSQL 資料庫，將新增通路、新拜訪資料直接保存。導入 Supabase Auth，第一線業務可利用手機 GPS 於抵達店家時進行經緯度比對「到店簽到打卡」。

3

階段三：多租戶 (SaaS) 與自動化通知擴展

長期規劃

在資料庫底層設計 `tenant_id`，實現多品牌藥廠獨立註冊之 SaaS 機制。整合 LINE Notify、Slack 等通訊軟體 Webhook，若偵測到通路急迫缺貨或效期亮紅燈，即時推播發送補貨催單提示給負責人。
